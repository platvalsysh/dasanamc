# 국제화 (i18n)

> **현재 상태**: 다국어 미지원. 한국어 하드코딩. 일부 영문 혼재 (예: 어드민 메뉴 빌더의 "Save Changes" vs "변경사항 저장"). 외주에서 영문/타국어 요구가 들어오면 마이그레이션 비용이 큼 — 본 문서는 도입 가이드.

## 권장 접근

- **단순/저비용**: 자체 `t()` 래퍼 + JSON 사전 (의존성 0)
- **표준/풍부한 기능**: [react-i18next](https://react.i18next.com)

외주 클라이언트가 다국어 한 번도 요구 안 할 가능성이 50% 이상이라면, _자체 래퍼로 시작_ → 진짜 필요할 때 i18next 로 이전.

## 단순 래퍼 도입 (의존성 0)

### 1. 사전 + 래퍼

```ts
// packages/core/src/ui/i18n.ts
import ko from "./locales/ko.json";
import en from "./locales/en.json";

const dicts = { ko, en } as const;
type Lang = keyof typeof dicts;

let current: Lang = "ko";
export function setLang(l: Lang) { current = l; }
export function t(key: string, vars?: Record<string, string | number>) {
  const raw = (dicts[current] as any)[key] ?? key;
  return vars
    ? raw.replace(/\{(\w+)\}/g, (_: string, k: string) => String(vars[k] ?? `{${k}}`))
    : raw;
}
```

### 2. 사용

```tsx
import { t } from "@repo/core/ui/i18n";

<button>{t("common.save")}</button>
<p>{t("greeting", { name: user.name })}</p>
```

### 3. 사전 파일 구조

```
packages/core/src/ui/locales/
  ko.json   # 공용 키 ("common.save", "common.cancel" ...)
  en.json
packages/module-board/src/locales/
  ko.json   # 모듈 특화 키 ("board.list.title" ...)
  en.json
```

## 마이그레이션 전략 (점진적)

1. 새로 작성하는 텍스트부터 `t()` 래핑
2. PR 단위로 1-2개 페이지씩 변환
3. ESLint 룰로 신규 하드코딩 차단 (warn 으로 시작):
   ```js
   "react/jsx-no-literals": ["warn", { ignore: [...] }]
   ```
4. 모든 페이지 변환 완료 시 룰을 `error` 로 승격

## 안티 패턴

- ❌ **DB 컬럼에 `name_ko`, `name_en` 분리** — 언어 추가 시 스키마 폭증. JSON 컬럼 + locale key 사용 권장
- ❌ **한 컴포넌트에 한국어/영어 혼재** — 빌더 메뉴 같은 사례 (현재 [packages/admin/src/routes/system/settings/menu.tsx](../../packages/admin/src/routes/system/settings/menu.tsx) 의 "Cancel"/"Save Changes" 와 "메뉴 항목 추가" 공존)
- ❌ **날짜/숫자 직접 포맷팅** — `Intl.DateTimeFormat` / `Intl.NumberFormat` 사용
- ❌ **메일/SMS 템플릿을 코드에 인라인** — DB 기반 템플릿 + 변수 치환
